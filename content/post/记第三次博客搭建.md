## 序
大概是为了在网安转专业面试的时候有点东西可以用来吹水，所以决定复活已经好久没有这么维护的Blog，又因为作者比较懒散，若还是和上两代Blog一样使用`WordPress+MySql+PHP`搭建，很可能因为携带维护服务器而丢失数据。将Blog Very.3.0部署在`Github Pages`上则可以很好的应对这个问题，即使懈怠维护，所有的文章、评论等都会保留在Github项目里，而不用担心服务器到期造成的损失

### 基本信息与副标题
先修改`author`、`email`和`description`三项填入自己的信息

副标题的`hugo-theme-reimu`支持打字机动画效果，可在`subtitle.typing`下配置语句：

```yaml
subtitle:
  typing:
    enable: true
    strings:
      - "若木秋光のBlog へようこそ！"
      - "ここから、始めましょう。一から…いいえ、ゼロから！"
      - "さあ、ゲームを始めよう。"
    typeSpeed: 100
    backSpeed: 50
    backDelay: 2000
```

三条句子会循环打出再删除，默认的速度等参数的效果就很不错。`text`字段是`typing.enable: false`时的静态回退文本，也一并修改

顺带开启了`banner_srcset`，配置不同分辨率的Banner图片，移动端加载更小的版本。`excerpt_link`也改成了`继续阅读`，更符合博客中文为主的思想

## Step 2:上传到Github Page
这一步是让`Github Actions`在每次push代码后自动构建Hugo站点并部署到`Github Pages`，省去手动操作

`hugo-reimu-template`里没有内置工作流文件，需要自己在`.github/workflows/`目录下新建一个`yaml`文件

整个工作流分两个job：`build`负责安装Hugo、构建站点并上传产物，`deploy`依赖`build`完成后将产物正式发布到`Github Pages`

`build` job里有几个细节需注意 ：

- `checkout`时加了`submodules: recursive`，因为主题是以子模块形式引入的，不加这个参数主题文件会拉不到
- `fetch-depth: 0`让git拉取完整历史，这样Hugo才能正确生成文章的`lastmod`时间
- Hugo需要安装`Extended`版本才能支持Sass/SCSS，从GitHub Release直接下载解压后加入`PATH`即可

```yaml
name: 构建并部署 Hugo 博客

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    env:
      HUGO_VERSION: 0.160.0
    steps:
      - name: 拉取代码
        uses: actions/checkout@v4
        with:
          submodules: recursive
          fetch-depth: 0

      - name: 安装 Hugo Extended
        run: |
          curl -sLJO "https://github.com/gohugoio/hugo/releases/download/v${HUGO_VERSION}/hugo_extended_${HUGO_VERSION}_linux-amd64.tar.gz"
          mkdir -p ~/.local/hugo
          tar -C ~/.local/hugo -xf "hugo_extended_${HUGO_VERSION}_linux-amd64.tar.gz"
          echo "$HOME/.local/hugo" >> $GITHUB_PATH

      - name: 配置 Pages
        id: pages
        uses: actions/configure-pages@v5

      - name: 构建站点
        run: hugo --gc --minify

      - name: 上传构建产物到github
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./public

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: 部署到 GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

工作流写完后还需要去仓库`Settings → Pages`把`Source`改为`Github Actions`，否则部署会静默失败什么提示都没有

期间踩了一个坑：最初构建命令写的是`hugo --gc --minify --baseURL "${{ steps.pages.outputs.base_url }}/"`，意图是让`configure-pages`这个Action动态传入`baseURL`，结果它传进来的是`http://`开头的地址，导致生成出来的HTML里所有绝对链接都是HTTP，在强制HTTPS的环境下全部mixed content报错。最后的修法是把`--baseURL`参数从构建命令里删掉，直接在`hugo.toml`里硬编码`baseURL = 'https://blog.rmqg.org/'`，问题解决。~~（找了好半天）~~

## Step 4:接入自己的域名
以上的操作都是在Github的域名`https://rmqg.github.io/`上完成，后续的操作即使用自己的域名指向Blog并做连接性能优化

### 绑定自定义域名
首先在Github仓库`Settings → Pages → Custom domain`填入`blog.rmqg.org`并保存，Github会自动在仓库根目录生成一个`CNAME`文件（内容就是一行`blog.rmqg.org`），并立刻触发一次构建部署把这个文件打进去（最终测试的时候发现访问`rmqg.github.io`实际上会重定向到`rmqg.org`，疑似因为浏览器缓存了弃用的DNS解析，换用无痕模式无此问题）

域名托管在`CloudFlare`，在DNS面板添加一条`CNAME`记录：名称填`blog`，目标填`rmqg.github.io`，右侧的云朵图标保持**灰色**（即Proxied关闭，仅DNS解析模式），保存后等DNS生效——生效时间一般几分钟到半小时不等，之后用`blog.rmqg.org`就能访问了

> [!TIP]
> 绑定自定义域名之后，访问`rmqg.github.io`会被Github自动302重定向到`blog.rmqg.org`，这是Github Pages的内置逻辑，与CloudFlare无关，不用单独配置任何跳转规则

回到Github Pages设置页，等待Github识别到DNS正常后，勾选`Enforce HTTPS`，Github会自动通过Let's Encrypt签发证书并在到期前自动续期。但是Github给的这张证书是Github代管的，无法导出密钥，后面接CDN时要注意这一点

> [!NOTE]
> 如果刚绑完域名发现`Enforce HTTPS`选项是灰的，等几分钟让Github签完证书就能勾了

### 曾经考虑过用CloudFlare橙云做加速
作者最初设想是直接开启CloudFlare橙云，让CF的CDN代理所有流量。测试后发现这条路经国内访问很慢，初次访问需要几十秒才能部分加载页面（部分CSS无法渲染），遂寻找其他的路经

所以CloudFlare这里始终保持灰云（仅DNS），把加速任务交给后面的国内CDN

### 发现加载很慢
上线之后发现Blog的首次加载时间奔一分钟去了，明显不正常，按F12打开开发者工具，Network面板一看，资源请求全卡在`Waiting (TTFB)`上，说明服务端响应本身就慢，不是客户端渲染的问题

排查思路：

1. 先排查DNS：`nslookup blog.rmqg.org`，解析结果正常指向Github的IP段，DNS没问题
2. 再直接测服务端：`ping rmqg.github.io`，丢包率高，延迟动辄两三百ms甚至超时——**根本原因找到了**：Github Pages的服务器在境外，国内访问延时与丢包率极高，每个资源文件的请求都有这个延迟，页面一共几十个资源，全加起来首屏加载当然慢到离谱

解法方向是在境内架一层CDN，用户的请求打到离他最近的国内CDN节点，节点有缓存就直接返回，缓存没命中才向源站（Github Pages）回源拉一次，后续其他用户访问同一资源就直接从节点取缓存，不再绕境外

### 尝试多吉云CDN（回源Github Pages）
`多吉云`由于有够用的免费CDN套餐，故选择`（作者没有恰米！）`

在多吉云控制台`CDN加速 → 域名管理 → 添加加速域名`，填入`blog.rmqg.org`，业务类型选`网页/下载`，然后配置回源信息：

- 回源协议：`HTTPS`
- 回源地址：`rmqg.github.io`（Github Pages的原始域名）
- 回源host：`blog.rmqg.org`

**回源host应当注意**：必须填`blog.rmqg.org`，不能填`rmqg.github.io`。原因是Github Pages用Host头来区分不同用户的站点，CDN向Github回源时如果Host头是`rmqg.github.io`，Github不知道要返回哪个仓库的内容，直接404。填成`blog.rmqg.org`之后Github才能正确路由到作者的仓库

同时，在CloudFlare的DNS里完成两件事：

1. 加一条`_cdnauth`的TXT记录完成多吉云的域名归属验证（值从多吉云控制台复制）
2. 把`blog`的CNAME目标从`rmqg.github.io`改成多吉云分配的`xxxxx.doge.com`格式CNAME接入点，灰云保持不动，切换完成

然后要给多吉云配HTTPS证书。需要注意的是，这里**不能用Github Pages的证书**，因为那张证书是Github代管的Let's Encrypt证书，私钥不在作者手中，根本没有可导出的`privkey.pem`可以上传给多吉云。多吉云提供了一个`证书无忧`功能，由多吉云自己代申请并续期证书，不需要手动上传，对作者这种情况正合适，在`HTTPS配置 → 证书无忧`里打开开关就行

不过证书无忧需要注意的是：

> 系统将于每天上午检查开启了证书无忧服务的域名证书到期情况，未配置证书的，或在证书到期前 10 天内的域名，将触发证书自动更新。

开启之后不会立刻签发，要等到第二天早上才触发申请。作者访问`https://blog.rmqg.org`一直无法正常访问，遂睡觉，第二天起来继续工作

第二天的测试效果：证书有了，HTTPS能访问了，有缓存的资源确实快了，但**缓存没有命中的时候依然很慢**——CDN回源要打到Github Pages服务器，Github那边的响应时间本身就长，CDN只是把"每个用户都慢一次"变成了"只有第一个用户慢一次"，对于作者这种访客极少的小Blog来说缓存命中率极低，实际改善效果相当有限

### 改用多吉云COS对象存储为源
治本的办法是把Blog的静态文件直接托管到多吉云自己的COS（对象存储）上，CDN以COS为回源，彻底绕开境外的Github Pages。COS服务器在国内，回源延迟几十毫秒量级，而且是内网互通，CDN的缓存冷启动速度也大幅提升

在多吉云控制台`对象存储 → 存储桶列表 → 新建存储桶`，地区选一个国内节点（作者因为肉身在广州，熟人也以南方居多，选择了广州节点），名称随意

创建完成后，在域名管理的基本配置-源站信息中编辑源站类型为云存储，选择刚刚创建的存储空间，确认修改

然后修改Github Actions工作流，在Hugo构建完、上传产物到Github Pages之后，加一步把`public/`目录同步上传到多吉云COS：

```yaml
- name: 上传到多吉云 COS
  uses: zu1k/dogecloud-cos-action@v0.1
  with:
    access_key: ${{ secrets.DOGE_ACCESS_KEY }}
    secret_key: ${{ secrets.DOGE_SECRET_KEY }}
    bucket: ${{ secrets.DOGE_BUCKET }}
    region: ${{ secrets.DOGE_REGION }}
    local_path: public
    remote_path: /
    clean: true
```

`DOGE_ACCESS_KEY`和`DOGE_SECRET_KEY`是多吉云账号的API密钥，在多吉云控制台`账号 → 密钥管理`里生成。`DOGE_BUCKET`是存储桶名称，`DOGE_REGION`是存储桶所在地区的代码（控制台里查），这四个值都作为Secret配置在仓库`Settings → Secrets and variables → Actions → New repository secret`里（逐一创建），**不要明文写在工作流文件里**，否则密钥会随公开仓库一起暴露出去

`clean: true`这个参数很重要：每次上传前会先清空COS里的所有旧文件，再上传新构建的内容。这样可以避免历史遗留文件（比如删掉的文章、改过路径的图片）继续留在COS里被访问到

工作流改完后push，Actions跑完后去多吉云COS的文件列表里确认，应该能看到所有的HTML、CSS、JS、字体、图片等文件都同步过来了。CDN从COS回源测试一切正常，首次加载时间从原来的几十秒缩短到两三秒以内，实测效果非常显著。Github Pages那边也保留着一份备份，可以在`https://github.com/rmqg/rmqg.github.io/settings/pages`取消`Custom domain`并x修改`.github/workflows/hugo.yaml`：

### 顺手处理了一下本机代理绕行问题
本地开发调试时挂着`Clash`代理，却发现访问`blog.rmqg.org`的流量也走了代理节点——先出国，再回来，等于完全绕过了国内的CDN，CDN加速白配了，而且还干扰了作者判断加速效果

Clash的规则列表是从上往下匹配的，匹配到第一条就停止， 不再检查后续规则。只需要在`/etc/clash/config.yaml`的`rules`字段**最前面**插入几条`DOMAIN-SUFFIX`直连规则，就能让这些域名的流量绕开代理直连：

```yaml
- DOMAIN-SUFFIX,dogecloud.com,DIRECT
- DOMAIN-SUFFIX,dogedns.com,DIRECT
- DOMAIN-SUFFIX,dogestore.com,DIRECT
- DOMAIN-SUFFIX,rmqg.org,DIRECT
```

前三条是多吉云相关域名（CDN接入点、DNS、COS存储），最后一条让作者自己的Blog域名也走直连。改完之后重载Clash配置（`sudo systemstl restart clash.server`重启），再访问`blog.rmqg.org`就走直连了，可验证CDN配置效果

后续测试发现作者使用`vercel`搭建的`meting-api`严重拖慢了网页加载，本着提升Blog稳定性和可维护性的原站，我使用了`injahow`官方维护的Api：`https://injahow.github.io/meting-api/`。修改完成后整个网页可以快速的访问
