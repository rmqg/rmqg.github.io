## 序
大概是為了在網安轉專業面試的時候有點東西可以用來吹水，所以決定復活已經好久沒有這麼維護的Blog，又因為作者比較懶散，若還是和上兩代Blog一樣使用`WordPress+MySql+PHP`搭建，很可能因為攜帶維護服務器而丟失數據。將Blog Very.3.0部署在`Github Pages`上則可以很好的應對這個問題，即使懈怠維護，所有的文章、評論等都會保留在Github項目裡，而不用擔心服務器到期造成的損失

## Step 2:上傳到Github Page
這一步是讓`Github Actions`在每次push代碼後自動構建Hugo站點並部署到`Github Pages`，省去手動操作

`hugo-reimu-template`裡沒有內置工作流文件，需要自己在`.github/workflows/`目錄下新建一個`yaml`文件

整個工作流分兩個job：`build`負責安裝Hugo、構建站點並上傳產物，`deploy`依賴`build`完成後將產物正式發布到`Github Pages`

`build` job裡有幾個細節需注意：

- `checkout`時加了`submodules: recursive`，因為主題是以子模塊形式引入的，不加這個參數主題文件會拉不到
- `fetch-depth: 0`讓git拉取完整歷史，這樣Hugo才能正確生成文章的`lastmod`時間
- Hugo需要安裝`Extended`版本才能支持Sass/SCSS，從GitHub Release直接下載解壓後加入`PATH`即可

```yaml
name: 構建並部署 Hugo 部落格

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    env:
      HUGO_VERSION: 0.160.0
    steps:
      - name: 拉取程式碼
        uses: actions/checkout@v4
        with:
          submodules: recursive
          fetch-depth: 0

      - name: 安裝 Hugo Extended
        run: |
          curl -sLJO "https://github.com/gohugoio/hugo/releases/download/v${HUGO_VERSION}/hugo_extended_${HUGO_VERSION}_linux-amd64.tar.gz"
          mkdir -p ~/.local/hugo
          tar -C ~/.local/hugo -xf "hugo_extended_${HUGO_VERSION}_linux-amd64.tar.gz"
          echo "$HOME/.local/hugo" >> $GITHUB_PATH

      - name: 配置 Pages
        id: pages
        uses: actions/configure-pages@v5

      - name: 構建站點
        run: hugo --gc --minify

      - name: 上傳構建產物至 GitHub
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./public

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: 部署至 GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

工作流寫完後還需要去倉庫`Settings → Pages`把`Source`改為`Github Actions`，否則部署會靜默失敗什麼提示都沒有

期間踩了一個坑：最初構建命令寫的是`hugo --gc --minify --baseURL "${{ steps.pages.outputs.base_url }}/"`，意圖是讓`configure-pages`這個Action動態傳入`baseURL`，結果它傳進來的是`http://`開頭的地址，導致生成出來的HTML裡所有絕對鏈接都是HTTP，在強制HTTPS的環境下全部mixed content報錯。最後的修法是把`--baseURL`參數從構建命令裡刪掉，直接在`hugo.toml`裡硬編碼`baseURL = 'https://blog.rmqg.org/'`，問題解決。~~（找了好半天）~~

## Step 4:接入自己的域名
以上的操作都是在Github的域名`https://rmqg.github.io/`上完成，後續的操作即使用自己的域名指向Blog並做連接性能優化

### 綁定自定義域名
首先在Github倉庫`Settings → Pages → Custom domain`填入`blog.rmqg.org`並保存，Github會自動在倉庫根目錄生成一個`CNAME`文件（內容就是一行`blog.rmqg.org`），並立刻觸發一次構建部署把這個文件打進去。在這次部署結束之前不要再push任何東西，等Actions跑完再操作

域名托管在`CloudFlare`，在DNS面板添加一條`CNAME`記錄：名稱填`blog`，目標填`rmqg.github.io`，右側的雲朵圖標保持**灰色**（即Proxied關閉，僅DNS解析模式），保存後等DNS生效，之後用`blog.rmqg.org`就能訪問了

> [!TIP]
> 綁定自定義域名之後，訪問`rmqg.github.io`會被Github自動302重定向到`blog.rmqg.org`，這是Github Pages的內置邏輯，與CloudFlare無關，不用單獨配置任何跳轉規則

回到Github Pages設置頁，等待Github識別到DNS正常後，勾選`Enforce HTTPS`，Github會自動通過Let's Encrypt簽發證書並在到期前自動續期。注意Github給的這張證書是Github代管的，無法導出密鑰，後面接CDN時要注意這一點

> [!NOTE]
> 如果剛綁完域名發現`Enforce HTTPS`選項是灰的，等幾分鐘讓Github簽完證書就能勾了

### 曾經考慮過用CloudFlare橙雲做加速
作者最初設想是直接開啟CloudFlare橙雲，讓CF的CDN代理所有流量。測試後發現這條路經國內訪問很慢，遂尋找其他方案

所以CloudFlare這裡始終保持灰雲（僅DNS），把加速任務交給後面的國內CDN

### 發現加載很慢
上線之後發現Blog的首次加載時間奔一分鐘去了，明顯不正常，按F12打開開發者工具，Network面板一看，資源請求全卡在`Waiting (TTFB)`上，說明服務端響應本身就慢

排查思路：

1. 先排查DNS：`nslookup blog.rmqg.org`，解析結果正常指向Github的IP段，DNS沒問題
2. 再直接測服務端：`ping rmqg.github.io`，丟包率高，延遲動輒兩三百ms甚至超時——**根本原因找到了**：Github Pages的服務器在境外，國內訪問延時與丟包率極高，頁面幾十個資源，全加起來首屏加載當然慢到離譜

解法方向是在境內架一層CDN，用戶的請求打到離他最近的國內CDN節點，節點有緩存就直接返回，緩存沒命中才向源站回源拉一次

### 嘗試多吉雲CDN（回源Github Pages）
`多吉雲`由於有夠用的免費CDN套餐，故選擇之。

在多吉雲控制台`CDN加速 → 域名管理 → 添加加速域名`，填入`blog.rmqg.org`，業務類型選`網頁/下載`，然後配置回源信息：

- 回源協議：`HTTPS`
- 回源地址：`rmqg.github.io`
- 回源host：`blog.rmqg.org`

**回源host應當注意**：必須填`blog.rmqg.org`，不能填`rmqg.github.io`。原因是Github Pages用Host頭來區分不同用戶的站點，填錯直接404

然後要給多吉雲配HTTPS證書。**不能用Github Pages的證書**，因為那張證書是Github代管的，私鑰不在作者手中。多吉雲提供了`證書無憂`功能，由多吉雲自己代申請並續期證書，在`HTTPS配置 → 證書無憂`裡打開開關就行

不過證書無憂需要注意：**系統只在每天上午定時檢查一次**，開啟後不會立刻簽發，要等到第二天早上。作者開啟之後等了大半天，當天下午訪問發現還是不通，折騰半天沒找出原因，最後去睡覺，第二天早上起來證書已經簽好了。~~（白白折騰了一下午）~~

測試效果：有緩存的資源確實快了，但**緩存沒有命中的時候依然很慢**，實際改善效果相當有限

### 改用多吉雲COS對象存儲為源
治本的辦法是把Blog的靜態文件直接托管到多吉雲自己的COS（對象存儲）上，CDN以COS為回源，徹底繞開境外的Github Pages

在多吉雲控制台`對象存儲 → 存儲桶列表 → 新建存儲桶`，地區選一個國內節點，名稱隨意

創建完成後，在域名管理的基本配置-源站信息中編輯源站類型為雲存儲，選擇剛剛創建的存儲空間，確認修改

然後修改Github Actions工作流，在Hugo構建完之後，加一步把`public/`目錄同步上傳到多吉雲COS：

```yaml
- name: 上傳至多吉雲 COS
  uses: zu1k/dogecloud-cos-action@v0.1
  with:
    access_key: ${{ secrets.DOGE_ACCESS_KEY }}
    secret_key: ${{ secrets.DOGE_SECRET_KEY }}
    bucket: ${{ secrets.DOGE_BUCKET }}
    region: ${{ secrets.DOGE_REGION }}
    local_path: public
    remote_path: /
    clean: true
```

四個Secret在倉庫`Settings → Secrets and variables → Actions → New repository secret`裡配置，**不要明文寫在工作流文件裡**，否則密鑰會隨公開倉庫一起暴露出去

工作流改完後push，Actions跑完後去多吉雲COS的文件列表裡確認，首次加載時間從原來的幾十秒縮短到兩三秒以內，實測效果非常顯著

### 順手處理了一下本機代理繞行問題
本地開發調試時掛著`Clash`代理，卻發現訪問`blog.rmqg.org`的流量也走了代理節點，CDN加速白配了
